# modification-guide — 修改治理與變更記錄

> 語言 / Language: **繁體中文** · [English](SKILL.en.md)

## 用途

任何對既有系統的修改或新功能,都先經過這裡:分析變更會動到什麼、產出明確的修改指引、同步更新結構文件、並留下一份變更記錄。目的是避免「直接改程式但文件與決策沒跟上」造成的架構漂移。

## 強制掛載規則(原始第 5 項需求)

**這是硬性規則:只要使用者在一個 session 中提出修改或新功能,AI 必須先載入本 skill 並依其流程進行,不可略過直接改碼。**

為什麼要強制:任何變更都可能牽動既有結構與先前的設計決策。若跳過治理直接改,容易出現與架構衝突的改動、遺漏的連帶修改、以及無人記得的「為什麼這樣改」。先過一次本流程,成本很低,卻能保住一致性與可追溯性。

判斷是否觸發(出現以下情況即視為變更,須掛載本 skill):

- 對已存在的功能/檔案/資料表要求調整、修正、擴充
- 新增一個功能到既有系統
- 重構、改名、搬移、刪除既有結構
- 任何會使現行結構文件失準的改動
- **驗收未通過交回修正**:`acceptance-verification` 產出未通過/部分通過,須回到本 skill 針對未達標項產出修正,再重新實作與驗收

例外:全新專案從零開始,走 `requirement-analysis` → `structure-design`,不適用本 skill。

## 工作流程

1. **讀基準**:讀 `docs/ai-guideline.md` 與 `docs/structure/*.md`,掌握現況。沒有這些文件時,先補建(回到對應 skill)或在記錄中註明缺漏。**若本次是因驗收未通過而進入,先讀對應的 `docs/acceptance/ACC-*.md`,把未達標項當作這次要修正的目標清單。**
2. **影響分析**:判斷這次變更會動到哪些模組、元件、資料、介面,以及連帶影響(相依方向、相容性、資料遷移)。
3. **產出修改指引**:用下方模板寫出具體、可執行的修改步驟,讓實作者(人或 AI)照做即可。
4. **調整結構文件**:若變更改動了結構,**同步更新** `docs/structure/` 對應檔案——結構文件必須永遠反映最新真實。
5. **留下變更記錄**:在 `docs/changes/` 新增一份記錄(見模板),寫清楚動機與取捨。因驗收未通過而修正者,在記錄的「關聯」欄連回該 ACC 報告。
6. **接驗收**:實作完成後,交給 `acceptance-verification` 依本次修改指引驗收。**若仍有未通過項,再次回到第 1 步,形成「修正 → 重新實作 → 重新驗收」的迴圈,直到全數通過或使用者明確接受。**

## 修改指引模板

寫入變更記錄中的「修改指引」段落,或視情況獨立成檔:

```markdown
## 修改指引
### 目標
<這次要達成什麼,對應哪條需求/問題>

### 影響範圍
| 受影響項 | 類型(目錄/邏輯/設計/資料) | 改動 | 連帶影響 |
|----------|------|------|----------|
| ... | 設計 | ... | ... |

### 修改步驟
1. ...
2. ...

### 結構文件更新
- [ ] docs/structure/<檔> 已同步:<改了什麼>

### 風險與回滾
- 風險:...
- 回滾方式:...
```

## 變更記錄模板

每次變更一份,檔名建議 `docs/changes/CHG-YYYYMMDD-NN.md`:

```markdown
# CHG-YYYYMMDD-NN — <變更標題>

- 日期:YYYY-MM-DD
- 類型:新功能 / 修正 / 重構 / 調整
- 提出者:<使用者>
- 關聯:<需求編號 / 先前變更 / 驗收報告>

## 動機
<為什麼要這個變更>

## 決策與取捨
| 決策 | 選項 | 為何這樣選 |
|------|------|------------|
| ... | A vs B | ... |

## 修改指引
<見上方修改指引模板>

## 結構異動摘要
<這次更新了哪些結構文件,重點是什麼>

## 狀態
草稿 / 已實作 / 已驗收(連結驗收報告)
```

## 撰寫要點

- **結構與記錄必須跟上程式**:本 skill 最大的價值是讓文件不落後。改了結構卻沒更新 `docs/structure/`,等於沒做完。
- **記錄「為什麼」,不只是「改了什麼」**:決策與取捨是未來最值錢的資訊。程式能看出改了什麼,看不出為何這樣選。
- **影響分析要往外看一層**:不只改動點本身,還要看相依它的東西會不會壞。
- **保持可追溯**:變更記錄連回需求編號與驗收報告,形成完整鏈路。

## 交付後

修改實作完成後,進入 `acceptance-verification`,以本次「修改指引」與「驗收條件」為基準確認結果。
