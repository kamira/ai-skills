# agent-hierarchy — 代理編制與階層

> 語言 / Language: **繁體中文** · [English](agent-hierarchy.md)

## 用途

把「多個 agent 一起做事」變成**有編制、有界線、可管理**的階層,而不是一堆 agent 各自亂跑。核心:**每個 agent 都被編號、被授予固定任務與範圍,不能做份外的事**;需要更多人力時可往下建子 agent,但**由上層主動管理、權限只能縮不能放**。

## 何時讀

- 一個任務要由多個 agent 分工(如 分析 / 實作 / 驗收)
- 某個 agent 工作量大,需要再分派子 agent 處理

## 編制原則

1. **編號 + 固定角色 + 固定範圍**:每個 agent 有唯一編號、固定角色、固定任務範圍(scope)與讀寫權限。建議命名反映階層:`A1`(分析)、`I1`(實作主)、`I1.1`/`I1.2`(I1 的實作子)、`V1`(驗收)。
2. **不得越權**:agent 只能做份內任務、只能讀寫被授予的範圍。**需要做範圍外的事 → 回報上層**,由上層決定(改派他人 / 擴大授權),**不可自行擴張**。
3. **遞迴授權,權限只能縮不能放**:子 agent 可在「自己被授予的範圍內」再建子 agent(孫),但:
   - 給下層的範圍**只能是自己範圍的子集**(權限不可放大);
   - 整條鏈(編號、角色、範圍、父子關係)都要登記在協調文件;
   - **由上層主動管理**:上層負責指派、追蹤、收斂,不能放生。
4. **上層職責**:指派任務 → 追蹤進度 → 確認與驗證子代理產出 → 收斂彙整 → 把錯誤入知識庫(見 agent-worklog)。

> 適用範圍:**單人或團隊皆適用**。個人用一個會開子代理的 AI,同樣可用此編制(分析/實作/驗收都由 AI 子代理擔任),你就是最上層管理者。

## 遞迴建置:現況與建議

- **巢狀子代理已支援**:子 agent 可再建子 agent;**最大深度 5**。所以遞迴授權路徑可用,不必退扁平。
- **但建議保持淺(2–3 層)**:層數越深,協調、追蹤、回溯成本越高、上層越難管理。需要更深時,通常代表任務該再拆分或重新設計,而非一路往下疊。
- **用 tools allowlist 控制「誰能 spawn」**:是否能再開子代理,由該 agent 的**工具授權**決定——只有需要分派的角色(如實作主 I1)才授予 `Agent`/spawn 工具;不需分派的角色(驗收 V1、純實作子 I1.x)**不給 `Agent` 工具**,從機制上擋住它再開子代理。這把「權限只縮不放」變成可由工具清單強制,而非只靠自律。
- **不變的治理原則**:不論層深,編號、固定範圍、不得越權、登記編制、上層主動管理、實作者不自驗都不變。

實務:超過 2–3 層前先想「是不是該把任務拆細或重畫邊界」;接近深度 5 應視為警訊。

## 典型鏈(實作情境範例)

```
A1 分析 ──► I1 實作主 ──► V1 獨立驗收
            ├─ I1.1 實作子(模組X)
            └─ I1.2 實作子(模組Y)
```

1. **分析 agent(A1)**:依需求產出 Guideline / 結構 / 影響分析——**只分析、不實作**。
2. **實作主 agent(I1)**:依 A1 的分析結果,把實作拆成子任務,分派給**實作子 agent(I1.1, I1.2…)**;每個子 agent 只做被分配的模組、只寫該模組範圍。
3. **I1 確認與驗證**每個子 agent 的產出(整合、跑測試、查一致性);全部子任務完成後彙整。
4. **全部實作完畢後**,交由**獨立驗收 agent(V1)**——V1 ≠ 實作鏈任何 agent、且**唯讀**——做多情境驗收,產出 ACC(見 independent-acceptance)。
5. 鐵律:**I1 不可自驗**(球員不可兼裁判);**V1 不可改碼**(發現問題→回 modification-guide 交實作鏈修)。

## 角色啟動規格(工具授權 allowlist)

每個 agent **啟動時依角色給定 tools allowlist**(最小權限)。這份規格是「當外部工具(python 等)呼叫/執行一個 AI 來擔任某角色」時的授權依據;**也適用於傳統的純 CLI / GUI 操作**——「可執行」涵蓋跑指令、跑測試、操作 GUI,不是只給 AI 用。

| 角色 | 可 spawn(`Agent` 工具) | 可寫程式/結構 | 可執行(CLI / 腳本 / GUI) | 說明 |
|------|------------------------|---------------|---------------------------|------|
| 分析 A1 | ✗ | ✗(只寫 docs/) | ✓(讀碼、跑分析工具) | 只分析,不實作 |
| 實作主 I1 | ✓(分派 I1.x) | ✓ 自己範圍 | ✓ | 需分派才給 `Agent` |
| 實作子 I1.x | ✗ | ✓ 自己模組 | ✓ | 不再分派 → 不給 `Agent` |
| **驗收 V1** | **✗(tools 不含 `Agent`)** | **✗(對被驗的碼/結構唯讀)** | **✓ 可跑測試/腳本/CLI/GUI 來驗證** | 只讀碼與條件、只寫自己的 ACC;**不可開子代理、不可改被驗的碼**,但**可執行**以進行多情境驗證 |

> **「唯讀」≠「不能執行」**:V1 的唯讀是指「不可修改被驗的程式與結構文件、不可 spawn 子代理」;它**仍可**用 python/CLI 跑測試、用 GUI 操作來驗證(這是驗收必需)。把「不含 `Agent` 工具」寫進 V1 的啟動規格,等於用工具清單在機制上保證它不會「邊驗邊改」或再開代理——比只靠自律可靠。

## 編制登記(協調文件)

在 `docs/coordination.md` 維護編制表,讓任何人/agent 一眼看懂誰在做什麼、誰管誰、給了什麼工具:

```markdown
# 代理編制
| 編號 | 角色 | 上層 | 任務 / 範圍 | 讀寫權限 | 工具授權 | 狀態 |
|------|------|------|-------------|----------|----------|------|
| A1   | 分析 | —    | 需求/結構/影響分析 | 讀碼;只寫 docs/ | 無 Agent;可執行分析 | 完成 |
| I1   | 實作主 | —  | 協調實作、整合 | RW src/ | 有 Agent(可分派);可執行 | 進行中 |
| I1.1 | 實作子 | I1 | 模組 X | RW src/x | 無 Agent;可執行 | 進行中 |
| I1.2 | 實作子 | I1 | 模組 Y | RW src/y | 無 Agent;可執行 | 完成 |
| V1   | 驗收 | —    | 多情境驗收 | 對碼唯讀;只寫 docs/acceptance | **無 Agent;可執行(跑測試/CLI/GUI)** | 待命 |
```

## 與既有流程的關係

- 延伸 `cross-agent` 的角色/權限/claim:本檔再加「**編號 + 階層 + 遞迴授權(只縮不放)+ 上層主動管理**」。
- 延伸 `agent-worklog`:每個編號 agent 進場先寫 worklog;遇錯回報上層;上層彙整錯誤入知識庫。
- 延伸 `independent-acceptance`:實作鏈(I1 及其子)做完 → 交獨立的 V1 驗收,實作者不自驗。
- 對應 ai-sdlc 階段:A1≈需求分析/結構設計,I1 鏈≈修改治理+實作,V1≈驗收。
