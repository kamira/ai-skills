# agent-hierarchy — 代理編制與階層

> 語言 / Language: **繁體中文** · [English](agent-hierarchy.md)

## 用途

把「多個 agent 一起做事」變成**有編制、有界線、可管理**的階層,而不是一堆 agent 各自亂跑。核心:**每個 agent 都被編號、被授予固定任務與範圍,不能做份外的事**;需要更多人力時可往下建子 agent,但**由上層主動管理、權限只能縮不能放**。

## 何時讀

- 一個任務要由多個 agent 分工(如 分析 / 實作 / 驗收)
- 某個 agent 工作量大,需要再分派子 agent 處理

## 編制原則

1. **編號 + 固定角色 + 固定範圍**:每個 agent 有唯一編號、固定角色、固定任務範圍(scope)與讀寫權限。建議命名反映階層:`A1`(分析)、`I1`(實作主)、`I1.1`/`I1.2`(I1 的實作子)、`V1`(驗收)。
2. **不得越權**:agent 只能做份內任務、只能讀寫被授予的範圍。**需要做範圍外的事 → 回報上層**,由上層決定(改派他人 / 擴大授權),**不可自行擴張**。
3. **遞迴授權,權限只能縮不能放**:子 agent 可在「自己被授予的範圍內」再建子 agent(孫),但:
   - 給下層的範圍**只能是自己範圍的子集**(權限不可放大);
   - 整條鏈(編號、角色、範圍、父子關係)都要登記在協調文件;
   - **由上層主動管理**:上層負責指派、追蹤、收斂,不能放生。
4. **上層職責**:指派任務 → 追蹤進度 → 確認與驗證子代理產出 → 收斂彙整 → 把錯誤入知識庫(見 agent-worklog)。

> 適用範圍:**單人或團隊皆適用**。個人用一個會開子代理的 AI,同樣可用此編制(分析/實作/驗收都由 AI 子代理擔任),你就是最上層管理者。

## 遞迴建置取決於 AI 機制(重要)

「子 agent 再建子 agent」能不能做,**取決於底層 AI 平台/機制是否允許巢狀 subagent**——有些機制只允許最上層派發子代理,子代理本身不能再派。因此:

- **若機制允許巢狀**:依上方「遞迴授權,權限只能縮不能放 + 上層主動管理」進行。
- **若機制不允許巢狀(只能單層派發)**:退而求其次,效果等價——
  - 由**最上層 agent 直接建立所有需要的 agent**(扁平編制),自己當每個人的上層;或
  - 該層 agent **自己把子任務依序做完**(不外派),完成後回報。
- **不論深淺,不變的是**:編號、固定範圍、不得越權、登記編制、上層管理、實作者不自驗。**只有「階層深度」受平台能力限制**,治理原則不變。

進場時應先確認當前機制是否支援巢狀;不確定就採扁平編制,最穩。

## 典型鏈(實作情境範例)

```
A1 分析 ──► I1 實作主 ──► V1 獨立驗收
            ├─ I1.1 實作子(模組X)
            └─ I1.2 實作子(模組Y)
```

1. **分析 agent(A1)**:依需求產出 Guideline / 結構 / 影響分析——**只分析、不實作**。
2. **實作主 agent(I1)**:依 A1 的分析結果,把實作拆成子任務,分派給**實作子 agent(I1.1, I1.2…)**;每個子 agent 只做被分配的模組、只寫該模組範圍。
3. **I1 確認與驗證**每個子 agent 的產出(整合、跑測試、查一致性);全部子任務完成後彙整。
4. **全部實作完畢後**,交由**獨立驗收 agent(V1)**——V1 ≠ 實作鏈任何 agent、且**唯讀**——做多情境驗收,產出 ACC(見 independent-acceptance)。
5. 鐵律:**I1 不可自驗**(球員不可兼裁判);**V1 不可改碼**(發現問題→回 modification-guide 交實作鏈修)。

## 編制登記(協調文件)

在 `docs/coordination.md` 維護編制表,讓任何人/agent 一眼看懂誰在做什麼、誰管誰:

```markdown
# 代理編制
| 編號 | 角色 | 上層 | 任務 / 範圍 | 讀寫權限 | 狀態 |
|------|------|------|-------------|----------|------|
| A1   | 分析 | —    | 需求/結構/影響分析 | docs/ 唯讀程式 | 完成 |
| I1   | 實作主 | —  | 協調實作、整合 | RW src/ | 進行中 |
| I1.1 | 實作子 | I1 | 模組 X | RW src/x | 進行中 |
| I1.2 | 實作子 | I1 | 模組 Y | RW src/y | 完成 |
| V1   | 驗收 | —    | 多情境驗收 | 唯讀 + 寫 docs/acceptance | 待命 |
```

## 與既有流程的關係

- 延伸 `cross-agent` 的角色/權限/claim:本檔再加「**編號 + 階層 + 遞迴授權(只縮不放)+ 上層主動管理**」。
- 延伸 `agent-worklog`:每個編號 agent 進場先寫 worklog;遇錯回報上層;上層彙整錯誤入知識庫。
- 延伸 `independent-acceptance`:實作鏈(I1 及其子)做完 → 交獨立的 V1 驗收,實作者不自驗。
- 對應 ai-sdlc 階段:A1≈需求分析/結構設計,I1 鏈≈修改治理+實作,V1≈驗收。
