# knowledge — 錯誤與修正指示知識庫治理

> 語言 / Language: **繁體中文** · [English](knowledge.md)

## 用途

知識庫是「讓 AI 不重犯同樣錯」的持久記憶,收兩類條目:

- **錯誤(error)**:執行中遇到的技術錯與解法(來自 agent-worklog)。
- **修正指示(DIR)**:**使用者明確的修正**——「不要用 X」「要這樣做」「這個之前就講過」。這**不是需求變更**(需求變更走 modification-guide 開 CHG),而是對「做法/風格/禁忌」的糾正,必須記下來避免下次再錯。
- **模式記錄(KN)**:**AI 觀察到的重複**——同樣的需求/目的在多次 CHG/需求中反覆出現。**自主建立** shallow record,靠「重複套用未被糾正」升 deep(見下方生命週期)。

位置:`docs/knowledge/`(建議 `errors.md`、`directives.md`——或合併成帶 INDEX 的 `knowledge.md`)。**進場讀索引、只載入 scope 內條目**(見下方檢索與 handshake)。

## 收錄修正指示(需求 3)

一收到使用者的修正指示(非需求變更),**立即寫入知識庫**,而不是只在當下改完就算:

```markdown
## DIR-<序號> — <一句話規則>
- 日期 / 分支:YYYY-MM-DD(UTC+0)/ <branch>
- 情境:<在什麼情況下適用>
- 規則:<該做什麼 / 不該做什麼>
- 原因:<為何(使用者的理由或推得的理由)>
- 來源:<使用者指示原文摘要>
- 信度:使用者確認 / agent 推斷   ← 這條規則由誰確立(見下方權重)
- 狀態:生效
```

## 更新既有指示(需求 4)

修正指示**再更新**時,**改寫既有條目,不是新增矛盾的一條**:

- 找到對應 DIR 條目 → 更新規則內容;把舊敘述移到「歷史」或標記取代(supersede),保留日期。
- 目標:同一規則永遠只有**一條現行版本**,避免知識庫自我矛盾(這本身也是一種漂移)。
- 若新指示與舊指示衝突,以新為準並記錄「取代 DIR-x(原因)」。

## 優先權重與衝突處理(需求 5)

知識庫是**高權重規則**,不是可有可無的參考——它代表「使用者已經教過、不該再犯」的事。

**信度與生命週期(同一把梯子,防污染)**:後續每個 session 都會照本檔行事,一條錯的或過度概括的條目會毒化整條下游。每條規則都在同一把梯子上:

| 層級 | 由誰確立 | 行為 | 推翻方式 |
|------|----------|------|----------|
| **shallow**(KN) | AI 觀察到重複——一個**假說** | 套用+**每次宣告**;每次套用未被糾正 `applied` +1 | 使用者一句話 |
| **deep**(KN) | 熬過使用的 shallow(`applied ≥ 3` 無糾正) | 預設套用、**不逐次宣告**,但 handshake ack 列出 | 使用者一句話 |
| **使用者確認**(DIR) | 使用者明說(或確認某 KN) | 有拘束力 | 下方三次確認 |

- **升級**:shallow → deep 條件為 `applied ≥ 3` 且無糾正(閾值可調;記升級日期)。KN → DIR **只能經由使用者實際陳述**——永不自我升級。
- **降級**:deep 被糾正 → 降回 shallow 或失效(記根因);糾正若含明確規則,反向確立為一條 DIR。
- agent 不得在沒有使用者實際陳述的情況下,把條目記成「使用者確認」。

## 自主觸發(shallow record)

不要等被糾正——**同樣的需求/目的第二次跨 CHG/需求出現,就觸發 shallow record**(閾值 2,可調)。可能跨 session,所以**證據與計數落在條目裡,不靠記憶**。檢查點在 CHG 收尾(見 modification-guide):本次變更的動機是否重複了先前某筆?

```markdown
## KN-<n> — <一句規則假說>
- tier:shallow / deep(+升級日期)
- 日期 / 分支:YYYY-MM-DD(UTC+0)/ <branch 或全分支>
- tags/scope:<模組 / 主題——小寫英文固定詞彙;這是檢索鍵>
- 觸發證據:<CHG-… 編號 / 需求提及(≥2 次)>
- 計數:seen <n> / applied <n> / last-applied <日期>
- 狀態:觀察中 / 生效 / 失效(原因)
```

## AI-friendly 結構與範圍檢索(INDEX)

沒有人該讀整份知識庫——那是目錄式思維。檔案**以 INDEX 開頭**;條目為帶錨點的小節,附 `tags/scope`:

```markdown
## INDEX(讀這裡,不是整份檔)
| id | tier | tags/scope | 一句規則 | 狀態 |
|----|------|-----------|----------|------|
| DIR-1 | 使用者確認 | 全分支 · api | 金額一律整數分 | 生效 |
| KN-2 | deep | report | 匯出一律 UTF-8 BOM | 生效(applied 5) |
| KN-3 | shallow | payment | 偏好 dry-run 先行 | 觀察中(seen 2 / applied 1) |
```

檢索規則(進場與派發皆同):讀 INDEX → 載入**全域條目 + tags 與當前 scope 相交的條目**——僅此而已。派發包用同一規則夾帶相符條目(見 handshake 範圍握手層)。

## 規模:一條目一檔(拆檔模式)

超過 **~30 條**,單檔就成了負債:並行 agent 下的單寫者熱點、跨分支的 merge 衝突磁鐵、`applied` 計數讓 diff 又髒又吵;手維護的 INDEX 開始與條目漂移。此時**拆檔**:

- **一條目一檔**:`docs/knowledge/entries/KN-004.md`、`entries/DIR-002.md`(檔名=id;檔名就是第一層過濾)。失效條目移 `entries/archive/`。
- **INDEX 變成生成物**:`docs/knowledge/INDEX.md`,由 `scripts/knowledge_index.py` 從條目 metadata 重生——**永不手改**(手維護的複本會漂移;生成物不會)。`--check` 驗新鮮;doc-integrity lint 雙向交叉驗「條目檔 ↔ INDEX id」。
- 檢索不變、與模式無關:讀 INDEX(單檔模式=檔首節;拆檔模式=`INDEX.md`)→ 只載 scope 內條目。

## AI-friendly 語言(正規化)

結構不夠,用字也算:

- **tags:小寫英文固定詞彙。** token 便宜、grep 穩定、跨模型(見平台中立);CJK 分詞邊界的模糊會傷檢索。
- **規則行:正規化。** 祈使句、一條一義、**可測試措辭**(「一律 X」——絕不是「盡量 X」)、不用含糊代名詞;規則容易被誤讀時,附一正一反例。
- **來源引文保留使用者原語言**(證據保真);規則行優先寫給機器讀。
- shallow → deep 升級時做一次**語言正規化**——把措辭洗乾淨是升級儀式的一部分。

- 動手前(handshake)先讀知識庫;規劃與實作都要遵守現行 directive。
- **當知識庫與使用者「新需求」衝突時**(新需求要做的,正好違反某條 directive):**不可自行取捨**(既不可默默照 knowledge、也不可默默照新需求)。改走**三次確認 + 影響揭露**:
  1. **第一次**:指出衝突——「這與 DIR-x『…』抵觸」,並**說明影響層面**(會動到什麼、為何當初這樣定、風險)。
  2. **第二次**:請使用者明確表示要覆寫該規則(而非隨口),再次列出受影響範圍。
  3. **第三次**:最終確認「確定覆寫 DIR-x?」得到肯定後才執行。
- 三次都確認覆寫 → 執行,並**更新知識庫**:把該 directive 標為「使用者於 <日期> 覆寫/放寬(原因)」或改寫規則;不是默默無視。
- 三次中任一未明確確認 → 維持原 directive,不執行衝突動作。

> 為什麼三次:高權重規則被推翻應該是慎重、可追溯的決定,不是一句話就翻案;三次確認 + 影響揭露讓使用者在充分知情下決定,並留下紀錄。

## 與既有流程的關係

- 與 agent-worklog:worklog 收「錯誤」,本檔把知識庫治理(含修正指示、更新、權重、衝突)講清楚;上層 agent 彙整錯誤與指示入此庫。
- 與 handshake:進場必讀知識庫。
- 與 modification-guide:**需求變更 → CHG**;**修正指示 → knowledge**。兩者不混。
- 與 branch-isolation:directive 標分支;跨分支引用比照需求規則(預設只採當前分支;通用規則可標「全分支」)。
