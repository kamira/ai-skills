# requirement-analysis — 需求分析產出 AI Guideline

> 語言 / Language: **繁體中文** · [English](requirement-analysis.md)

## 用途

把使用者的需求(往往零散、口語、隱含假設)轉成一份結構化的 **AI Guideline**。這份文件是後續所有階段(結構設計、修改、驗收)的單一依據,讓 AI 不必每次重新猜測使用者要什麼。

## 使用時機

- 收到新專案或新的大功能需求
- 需求模糊、口語、夾雜多個目標,需要先釐清
- 使用者問「這個要怎麼規劃/怎麼做」
- 在進入 `structure-design` 之前

如果是對「既有系統」提出修改或新增功能,改用 `modification-guide`(它會在需要時回頭引用本階段的 Guideline)。

## 工作流程

1. **讀既有脈絡**:若專案已有 `docs/ai-guideline.md`,先讀,避免重複或衝突。
2. **釐清,不臆測**:對關鍵未知處主動提問——目標、使用者、規模、限制、期限、技術偏好、驗收標準。一次問重點,不要一次丟一堆問題淹沒使用者。
3. **歸納需求**:把對話拆成功能需求與非功能需求,標出假設與待確認項。
4. **產出 Guideline**:用下方模板寫出 `docs/ai-guideline.md`。
5. **交回確認**:請使用者確認 Guideline,再進入結構設計。

## 釐清重點清單

問問題時涵蓋(但不機械式照唸):

- **目標**:要解決什麼問題?成功長什麼樣?
- **使用者/利害關係人**:誰用?誰決策?
- **範圍**:這次要做什麼、明確不做什麼(避免範圍蔓延)。
- **限制與假設**:期限、預算、既有系統、合規、技術偏好。
- **非功能需求**:效能、安全、可維護性、相容性、規模。
- **驗收條件**:怎樣算完成?可量測的標準。

## AI Guideline 模板

產出 `docs/ai-guideline.md`,固定使用此結構:

```markdown
# AI Guideline — <專案/功能名稱>

- 專案:<專案代號 / 名稱>   ← 跨專案時必填,讓本文件可明確歸屬
- 版本:v1.0
- 日期:YYYY-MM-DD
- 狀態:草稿 / 已確認

## 1. 背景與目標
<為什麼做、要解決的問題、成功定義>

## 2. 範圍
### 納入
- ...
### 不納入(明確排除)
- ...

## 3. 利害關係人
| 角色 | 關注點 |
|------|--------|
| ... | ... |

## 4. 功能需求
| 編號 | 需求 | 優先級 (P0/P1/P2) | 備註 |
|------|------|------|------|
| FR-1 | ... | P0 | ... |

## 5. 非功能需求
| 類別 | 要求 |
|------|------|
| 效能 | ... |
| 安全 | ... |
| 可維護性 | ... |
| 相容性/規模 | ... |

## 6. 限制與假設
- 限制:...
- 假設:...
- 待確認:...

## 7. 驗收條件
- [ ] <可量測、對應某條需求的標準>

## 8. AI 開發約定
<後續 AI 開發時要遵守的原則:命名、技術選型方向、文件存放位置、與哪些 skill 搭配等>
```

## 撰寫要點

- **驗收條件要可量測**:「快」不行,「列表頁 95 百分位 < 500ms」可以。每條盡量對應到某條功能需求。
- **明確寫出「不納入」**:範圍的邊界比範圍本身更能防止後續爭議。
- **標記待確認**:不確定的別硬填,放進「待確認」,讓使用者補。
- **第 8 節是給未來的 AI 看的**:把這次專案特有的約定寫清楚,後續階段才有依據。

## 交付後

Guideline 經使用者確認後,進入 `structure-design`,依 Guideline 產出系統結構。
