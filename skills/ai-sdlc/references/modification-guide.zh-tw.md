# modification-guide — 修改治理與變更記錄

> 語言 / Language: **繁體中文** · [English](modification-guide.md)

## 用途

任何對既有系統的修改或新功能,都先經過這裡:分析變更會動到什麼、產出明確的修改指引、同步更新結構文件、並留下一份變更記錄。目的是避免「直接改程式但文件與決策沒跟上」造成的架構漂移。

## 強制掛載規則(原始第 5 項需求)

**這是硬性規則:只要使用者在一個 session 中提出修改或新功能,AI 必須先載入本 skill 並依其流程進行,不可略過直接改碼。**

為什麼要強制:任何變更都可能牽動既有結構與先前的設計決策。若跳過治理直接改,容易出現與架構衝突的改動、遺漏的連帶修改、以及無人記得的「為什麼這樣改」。先過一次本流程,成本很低,卻能保住一致性與可追溯性。

判斷是否觸發(出現以下情況即視為變更,須掛載本 skill):

- 對已存在的功能/檔案/資料表要求調整、修正、擴充
- 新增一個功能到既有系統
- 重構、改名、搬移、刪除既有結構
- **回滾/撤銷先前的變更**(含 git revert):回滾本身也是一筆變更——開一份 CHG 連回原 CHG,並把結構文件改回與回滾後狀態一致
- 任何會使現行結構文件失準的改動
- **驗收未通過交回修正**:`acceptance-verification` 產出未通過/部分通過,須回到本 skill 針對未達標項產出修正,再重新實作與驗收

例外:全新專案從零開始,走 `requirement-analysis` → `structure-design`,不適用本 skill。

## 工作流程

1. **讀基準**:讀 `docs/ai-guideline.md` 與 `docs/structure/*.md`,掌握現況。沒有這些文件時,先補建(回到對應 skill)或在記錄中註明缺漏。**若本次是因驗收未通過而進入,先讀對應的 `docs/acceptance/ACC-*.md`,把未達標項當作這次要修正的目標清單。**
2. **影響分析**:判斷這次變更會動到哪些模組、元件、資料、介面,以及連帶影響(相依方向、相容性、資料遷移)。**同時查假設**:掃過受影響範圍內先前 CHG 的決策表——若本次變更推翻了某條記錄在案的前提假設,該決策必須重新評估(併入本 CHG 或另開一筆),不可默默放著。
3. **產出修改指引**:用下方模板寫出具體、可執行的修改步驟,讓實作者(人或 AI)照做即可。實作過程中,**每完成一步立刻打勾**——萬一 session 中途掛掉,下一棒用未勾步驟對帳 working tree(見 handshake),不必用猜的。
4. **調整結構文件 + 修正 Guideline**:若變更改動了結構,**同步更新** `docs/structure/` 對應檔案——結構文件必須永遠反映最新真實。若變更也動到需求/範圍/驗收條件,**一併修正 `docs/ai-guideline.md` 並升版**(過時的 Guideline 會誤導後續)。
5. **留下變更記錄**:在 `docs/changes/` 新增一份記錄(見模板),寫清楚動機與取捨。因驗收未通過而修正者,在記錄的「關聯」欄連回該 ACC 報告。
6. **確認閘(動碼之前)**:把摘要給使用者確認才實作——動機、影響範圍、**代決事項**(凡從 Guideline/文件/使用者指示推導不出的決策)、風險分級。使用者在此**覆核風險欄**(見分級:高風險清單命中不接受自評降級)。**預授權**:使用者可對某類變更豁免逐次確認(「這類直接做」)——記為 knowledge directive,且必須寫**窄**(情境+邊界;邊界模糊時回退為問)。使用次數計入治理健康度;闖禍的類別**自動失效**(見 CHG-lite)。單人 session 中,AI 發現同一類變更被重複確認時,應**主動建議**預授權。自主連跑走 halt 契約(autonomy)而非本互動閘——同一意圖、兩個通道。
7. **當場驗收(不可交棒)**:實作完成後,**在同一輪內**立刻依 `acceptance-verification` 產出對應的 `docs/acceptance/ACC-*.md`,並把本 CHG 狀態更新為「已驗收」並連結該 ACC。**不可只標「待驗收」就結束**——跨 session 開發下,下一個 session 來的是新需求、不會有人回頭補驗收,驗收一旦交棒就會永遠懸著。**若仍有未通過項,回到第 1 步,形成「修正 → 重新實作 → 重新驗收」的迴圈,直到全數通過或使用者明確接受。**

## 修改指引模板

寫入變更記錄中的「修改指引」段落,或視情況獨立成檔:

```markdown
## 修改指引
### 目標
<這次要達成什麼,對應哪條需求/問題>

### 影響範圍
| 受影響項 | 類型(目錄/邏輯/設計/資料) | 改動 | 連帶影響 |
|----------|------|------|----------|
| ... | 設計 | ... | ... |

### 修改步驟(每完成一步立刻打勾——勾選狀態就是中斷後的續作點)
- [ ] 1. ...
- [ ] 2. ...

### 結構文件更新
- [ ] docs/structure/<檔> 已同步:<改了什麼>

### 風險與回滾
- 風險:...
- 回滾方式:...
```

## 變更記錄模板

每次變更一份,檔名建議 `docs/changes/CHG-YYYYMMDD-NN.md`:

```markdown
# CHG-YYYYMMDD-NN — <變更標題>

- 專案:<專案代號 / 名稱>   ← 跨專案時必填;同時動多個專案時,變更編號建議帶專案前綴(如 PROJ-CHG-…)
- 分支(Branch):<branch>   ← 多分支時必填;只引用同分支需求/驗收(見 branch-isolation)
- 日期:YYYY-MM-DD(UTC+0)
- 類型:新功能 / 修正 / 重構 / 調整
- 提出者:<使用者>
- 實作者:<人 / agent id>   ← 用於「驗收者 ≠ 實作者」的身分比對
- 風險分級:高 / 中 / 低(見下方分級指引;驅動驗收嚴格度、CI 門檻與自主停點)
- Autonomy:(選用)auto / halt   ← 覆寫自主執行停點(只准加嚴;見 autonomy)
- Commit/PR:<hash / PR 連結>   ← 收尾時填;見下方「commit 粒度」
- Skill: ai-sdlc v<X.Y>   ← 本記錄依哪版慣例寫成(新規則只往後適用;見 doc-integrity)
- 關聯:<需求編號 / 先前變更 / 驗收報告>

## 動機
<為什麼要這個變更>

## 決策與取捨
| 決策 | 選項 | 前提假設(此決策建立在什麼之上) | 為何這樣選 |
|------|------|----------------------------------|------------|
| ... | A vs B | ... | ... |

## 修改指引
<見上方修改指引模板>

## 結構異動摘要
<這次更新了哪些結構文件,重點是什麼>

## 狀態
草稿 / 已實作 / 暫停(原因+恢復條件) / 已驗收(連結驗收報告)
```

**commit 粒度(commit 錨定)**:程式碼、本 CHG、對應 ACC 落在**同一個 commit / PR**,且 **commit message 帶 CHG 編號**(如 `CHG-20260702-02: …`)。這是歷史可對帳的根基——handshake 的 commit 掃描會把「沒有引用任何 CHG 的 commit」標為未治理工作。收尾時回填標頭的 `Commit/PR:` 欄。

**PR / squash / rebase 工作流**:活得過歷史改寫的是 **message 與 PR 號**,不是 hash——所以 message 裡的 CHG 編號是主錨點,hash 是盡力而為。squash merge:squash commit **必須帶 CHG 編號**(放 PR 標題,或確保進 squash message);trunk 以 squash commit 為粒度掃描,feature 分支在 squash 前仍逐 commit 掃。rebase / force-push:編號比對不怕改寫;收尾時 `Commit/PR:` 回填 **trunk commit / PR 號**(穩定),不要填 rebase 前的 hash。

**暫停變更(需求交錯)**:新需求插入、打斷進行中的 CHG 時,不可放著曖昧——把狀態改為**暫停**並寫明原因與恢復條件。暫停中的 CHG 每次進場都會被列出,由人有意識地恢復或收掉;暫停是合法的 WIP,和「懸空驗收」(已實作卻沒驗收)不同——後者仍必須先補完。

## 風險分級與對應嚴格度

每筆變更先評風險,讓**治理嚴格度與風險匹配**(低風險別過度、高風險別輕放):

| 等級 | 典型情況 | 要求的嚴格度 |
|------|----------|--------------|
| **高** | 動到資料模型/遷移、認證授權、金流、刪除/不可逆、跨模組介面、安全相關 | **全席審議會**(見 review-panel)+ 必須**獨立驗收**(驗收者 ≠ 實作者)+ **多情境** + **跑迴歸(受影響範圍)**;CI 做**身分檢查**;走完整 pipeline;回滾計畫必填 |
| **中** | 既有功能行為調整、新增非破壞性端點/欄位 | 三席審議(risk/impact/drift);至少獨立驗收或完整測試;**跑迴歸(受影響範圍)**;結構同步;pipeline 門檻 |
| **低** | 文案、註解、樣式、純內部重構且測試涵蓋 | 可自驗 + 測試必綠;pre-commit 即可 |

判級就高不就低:落在邊界時,以較高等級處理。風險寫進 CHG 標頭,驅動後續驗收與 CI 門檻。

**分級不是純自評**:高風險清單(資料模型/遷移、認證授權、金流、刪除/不可逆、跨模組介面、安全)命中即**高**,不論 AI 自己怎麼評——不接受自評降級;且使用者會在確認閘覆核風險欄。沒被考慮到的情況,不能只因為 AI 自評「低」就滑過所有閘門。

## 低風險輕量記錄(CHG-lite)

記帳成本也要隨風險縮放——改個錯字也要填整份模板,等於教人繞過流程。**低風險**變更用一屏的 CHG 即可,編號與錨定規則不變:

```markdown
# CHG-YYYYMMDD-NN — <標題>
- 日期:YYYY-MM-DD(UTC+0)| 分支:<branch> | 風險分級:低 | 實作者:<id>
- 動機:<一句話>
- Commit/PR:<hash / PR>
## 狀態
已驗收——自驗(低風險):<一行可重跑證據,如 `pytest tests/x -q` 全綠>
```

- **資格是白名單,不是感覺**:文案/註解、樣式、純文件修訂、有測試覆蓋的純內部重構。動到結構、資料、介面的一律出局;白名單外→完整模板,再小也一樣。
- 內嵌自驗行**只在低風險**可取代獨立 ACC(分級表本就允許低風險自驗)——必須含證據;lint 認可(標了自驗的低風險 CHG 豁免 ACC 檔要求)。
- **闖禍即強制升級**:lite 變更事後被迴歸/驗收抓到弄壞東西 → 補開**完整 CHG**、根因入 knowledge,且涵蓋該類別的預授權**自動失效**,直到使用者重新授權。
- 中/高風險一律完整模板 + 獨立 ACC(+ 依風險開審議席,見 review-panel)。

## 緊急通道(expedited path)

production 掛了、資料正在流失時,「先治理再動手」會擋住救火——這正是大家整個繞過流程的時刻。所以要有**合法的緊急車道**:

- **觸發條件**:由人明確宣告緊急(production 事故、進行中的資料損失/安全暴露)。「我趕時間」不算。
- **當下**:先修。過程留最小痕跡——worklog 一行(「緊急:壞了什麼、我在改什麼」)。
- **事後(24h 內)**:補記**追溯 CHG**(`類型:緊急 / 追溯`,連結事故與 commits)+ 對應 ACC,並同步結構文件。緊急車道**延後**記帳,從不免除記帳。
- **稽核**:追溯 CHG 有明確標記,健康度度量與回顧看得到使用頻率——常用代表正常流程太慢,這本身就是一個發現。
- **違規的定義是「不補記」,不是「用了車道」。**

與 autonomy 的關係:人明確宣告緊急,即視為停點關卡所等的人為核准(見 autonomy);永遠停點動作仍由人執行/在場,不得靜默代行。

## 撰寫要點

- **結構與記錄必須跟上程式**:本 skill 最大的價值是讓文件不落後。改了結構卻沒更新 `docs/structure/`,等於沒做完。
- **記錄「為什麼」,不只是「改了什麼」**:決策與取捨是未來最值錢的資訊。程式能看出改了什麼,看不出為何這樣選。
- **影響分析要往外看一層**:不只改動點本身,還要看相依它的東西會不會壞。
- **保持可追溯**:變更記錄連回需求編號與驗收報告,形成完整鏈路。

## 交付後

修改實作完成後,進入 `acceptance-verification`,以本次「修改指引」與「驗收條件」為基準確認結果。
