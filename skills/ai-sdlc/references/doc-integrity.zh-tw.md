# doc-integrity — 文檔抗漂移與驗證

> 語言 / Language: **繁體中文** · [English](doc-integrity.md)

## 用途

ai-sdlc 把 `docs/` 當作後續工作的依據(**單人或團隊皆然**)——所以**文件一旦與真實(程式碼)或彼此漂移,後續決策就建立在錯的基礎上**。漂移不是團隊才有的問題:單人/單一 agent 也會「改了程式卻忘了更新文件」,只是團隊/多 agent 時更嚴重、更難察覺。本檔確保文件不只是「被寫出來」,而是**持續被驗證、保持抗漂移**。

## 何時讀

- 要確認既有 `docs/` 是否仍可信(尤其接手別人/別的 agent 的專案)
- 每次變更收尾時(連同當場驗收一起檢查文件一致)
- agent 進場做 Session 啟動檢查時

## 兩類一致性(都要顧)

### 1. 程式 ↔ 文件(縱向)
- 改了結構性程式(模組、資料模型、介面/API),`docs/structure/` 必須同步;反之,文件描述的實體/模組/介面,要在程式中找得到。
- 驗收條件(Guideline §7)要對得上實際行為與測試。

### 2. 文件 ↔ 文件(橫向)
- 四種結構彼此對得上:邏輯的模組 ↔ 目錄的資料夾 ↔ 設計的元件 ↔ 資料的實體,名稱與邊界一致。
- Guideline 的每條 FR 都能往下追溯到結構與某份 ACC;每份 CHG 都能往回連到 FR、往前連到 ACC。

## 抗漂移觸發點

不是一次性,而是嵌在流程裡反覆檢查:
- **變更收尾**:CHG 收尾時,一併確認結構文件已同步(對應 modification-guide「文件即真實」)。
- **進場**:Session 啟動檢查時順手掃文件一致性,發現漂移先修再做新事。
- **CI(選用)**:用 ci-cd 的「結構同步檢查」把它自動化成 PR 門檻。

## 驗證方式

- **可程式化的就寫檢查腳本**(優先,因為可重複、無偏誤):
  - 本 skill 附 **`scripts/doc_integrity_check.py`**:檢查「改了結構性程式卻沒同步 `docs/structure/`」與「已實作的 CHG 沒有對應 ACC」,不過就回非零碼。接 pre-commit / CI 即可把「靠遵守」變「機器擋」:`python3 scripts/doc_integrity_check.py --staged`。
  - 文件中提到的實體/模組名,用 grep 在程式中找不到 → flag(可能文件過時或命名漂移)。
  - 每條 FR 在結構/ACC 出現過。
- **無法程式化的由獨立 agent 審閱**:語意是否仍正確、決策理由是否仍成立——交給非作者的 agent 看(見 independent-acceptance)。

## 文件驗證清單

每份文件應滿足:
- [ ] 有版本與日期(可看出新舊)
- [ ] 可追溯:連得到對應的 FR / CHG / ACC 編號
- [ ] 與最新程式一致(縱向)
- [ ] 與其他結構文件一致(橫向)

## 發現漂移時

**不要默默改文件**——回 `modification-guide` 補一筆「文件同步」變更(或在當前 CHG 註明),寫清楚漂移內容與修正,維持留痕。漂移本身就是一個值得記錄的訊號:它代表先前某次變更沒把文件帶上。
