# agent-worklog — 子代理工作日誌與錯誤知識庫

> 語言 / Language: **繁體中文** · [English](agent-worklog.md)

## 用途

當一個 agent 派發子代理(subagent)執行任務時——**這不限團隊,單人用一個會開子代理的 AI 也會發生**——用工作日誌把「正在做什麼、遇到什麼錯、怎麼解的」全部留痕,並由上層 agent 最後把錯誤統整進**知識庫**,讓未來的任務不再重蹈覆轍。記憶會中斷、會壓縮、會換手;日誌與知識庫是它的持久替代。

## 何時讀

- 你(上層/orchestrator agent)準備派子代理做事
- 你是被派的子代理,開始執行任務之前

## 子代理規則(被派的 agent)

1. **執行前先寫**:動手前,先在工作日誌寫下「我要做什麼」——任務、鎖定範圍、預期產出、角色與讀寫權限。這樣一旦中斷/壓縮/換手,別人或未來的你能看到你做到哪、原本要做什麼。
2. **遇錯記錄並說明再繼續**:發生錯誤時,記下「**什麼錯誤 + 根因 + 如何解決**」,解決後才繼續;**不可靜默吞錯或假裝沒事**。若無法解決,記下卡點與已嘗試的方案,回報上層、不要硬推。
3. **完成時回報**:回報產出 + **本次遇到的錯誤清單**(供上層彙整);若無錯誤也明說。

## 上層 agent 規則(orchestrator / parent)

- **派工前指定角色與讀寫權限**:每個子代理都要有明確角色與可讀/可寫範圍(見 cross-agent、independent-acceptance);驗收類子代理唯讀。
- **收集並統整錯誤**:把所有子代理回報的錯誤,最後統整進**知識庫** `docs/knowledge/errors.md`——去重、歸納共通模式、寫下「錯誤 → 根因 → 解法 → 預防」。
- **知識庫是累積資產**:每個 agent 進場(配合 Session 啟動檢查)**先讀知識庫**,避免重犯已知錯誤。

## 檔案位置

- **工作日誌(短期、執行期)**:`docs/worklog/`(每個子代理一筆,或一個 append-only 檔);並行時可併入協調文件。任務收尾後可清理或封存。
- **錯誤知識庫(持久、跨任務)**:`docs/knowledge/errors.md`。

## 知識庫模板

```markdown
# 錯誤知識庫
| 日期 | 情境 / 任務 | 錯誤 | 根因 | 解法 | 預防 |
|------|------------|------|------|------|------|
| YYYY-MM-DD | <在做什麼> | <錯誤訊息/現象> | <為什麼會發生> | <怎麼解的> | <下次如何避免> |
```

## 工作日誌模板(單筆)

```markdown
## <子代理/任務名> — YYYY-MM-DD HH:MM
- 角色 / 讀寫權限:<role / RW scope>
- 正在做:<任務、鎖定範圍、預期產出>
- 進度:進行中 / 完成 / 受阻
- 遇到的錯誤:
  - 錯誤:... | 根因:... | 解法:...(或:無法解決,卡在 X,已試 Y → 回報上層)
```

## 與既有流程的關係

- 「執行前先寫」呼應 doc-integrity 與「不倚賴記憶,以文件為準」:狀態落在文件而非腦中,抗中斷/壓縮/換手。
- 錯誤知識庫是「決策(CHG)/驗收(ACC)」之外的第三種留痕:不只記「做了什麼決策」,也記「踩了什麼坑、怎麼避免」。
- 上層彙整是 parent agent 的職責;**單人也適用**——此時你自己就是那個 parent。
