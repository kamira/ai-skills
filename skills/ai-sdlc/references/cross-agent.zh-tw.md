# cross-agent — 跨 agent 協作與交接機制

> 語言 / Language: **繁體中文** · [English](cross-agent.md)

## 用途

當同一專案由多個 AI agent、或同一 agent 跨多個 session 接力/並行開發時,維持一致、不衝突、可交接。核心是把 `docs/` 從「記憶」升級為「agent 之間的協作媒介」。

## 何時讀

- 工作會由不同 session / 不同 agent 接續(跨會話累加開發)
- 多個 agent 可能同時動同一專案
- 任何「換手」情境(交接給別人或未來的自己)

## 核心原則:docs/ 是 agent 之間唯一的事實來源

不靠對話記憶在 agent 間傳遞狀態——對話記憶不可跨 agent、且會被壓縮。每個 agent **進場先讀 `docs/`**(呼應本 skill 的「Session 啟動檢查」與原則「不倚賴記憶,以文件為準」)。能被下一個 agent 接續的前提,是上一個 agent 把狀態完整寫進了文件。

## 模式 A:順序交接(一次一個 agent)

這是已驗證有效的常態(對應 loop 測試的跨 session 情境)。

- **進場**:做 Session 啟動檢查——掃 `docs/changes/` 是否有狀態≠已驗收、`docs/acceptance/` 是否缺對應 ACC;有就先補驗收,再開始新工作。
- **收尾(交接前)**:當場驗收、回填 CHG 狀態為「已驗收」、結構文件同步——把一個「乾淨、可續」的狀態留給下一棒。
- **交接檢查清單**:離場前確認
  - [ ] docs 與程式一致(結構文件反映最新真實)
  - [ ] 無懸空階段(每個 CHG 都有對應 ACC)
  - [ ] 最新決策與待確認項已寫進 CHG / Guideline
  - [ ] 下一步在記錄中講清楚(下一棒讀文件就能接)

## 模式 B:並行多 agent(同時動同一專案)

風險:互相覆蓋、結構文件打架、重複或衝突的變更、CHG 編號相撞。機制:

- **工作認領(claim)**:動工前先在協調文件(見下)宣告「我是什麼**角色**、要做 X、鎖定範圍 Y、我的**讀寫權限**」,標 owner + 開始/**最後更新**時間(UTC+0)+ 狀態(進行中)。先佔再做。
- **claim 租約與停滯接管(防死鎖)**:claim 是**租約,不是永久佔有**——每次更新 worklog/協調文件就刷新最後更新時間。逾期(建議 24h,以 UTC+0 時鐘判定,團隊可調)且**worklog/協調文件無更新、也無對應 git 活動**的 claim 視為**停滯**(owner 多半中途掛掉):不要無限等待——在協調文件標 `停滯,由 <id> 於 <time> 接管`(不刪原記錄),依 handshake 對帳其 worklog + working tree 判斷做到哪,再接手或釋放範圍。
- **角色與讀寫權限**:每個 agent 都要有明確角色(實作者 / 驗收者 / 整合者 / 審閱者…)與**可讀/可寫範圍**。最小權限原則:只給完成該角色所需的寫入範圍。例如驗收者**唯讀**(不可改被驗的碼);實作者只可寫自己 claim 的範圍;結構文件由負責該模組者寫。這既是並行防誤改,也是獨立驗收的前提。
- **範圍邊界**:claim 切到**不重疊**的模組/檔案範圍。範圍重疊時,後到者等待或與前者協調,不可硬上。
- **單寫者規則**:同一份結構文件、同一模組,同一時間只由一個 agent 寫,避免 race condition。
- **實體隔離(worktree)**:claim 只是**邏輯**鎖——兩個 agent 在**同一個 working tree** 裡,未提交的變更照樣互相看見、互相踩。真正並行時,每個 agent 用自己的 `git worktree`(或 clone);合回去走正常 merge 流程+整合驗收。claim=誰可以做;worktree=在哪裡做。
- **CHG 編號先佔**:CHG 用「日期+序號」,claim 時就先佔號(如 CHG-YYYYMMDD-03),避免兩個 agent 撞號。
- **完成即釋放**:做完自己的 CHG + ACC 後,更新協調文件把 claim 標為已完成、釋放範圍。
- **整合驗收**:若兩個並行變更動到同一結構或彼此相依,需有一次「整合驗收」確認彼此相容(由其中一個 agent 或一個專責整合 session 做),不可各自綠燈就當沒事。
- **衝突處理**:若發現自己的假設與他人已提交的 CHG 衝突 → **停**,讀對方 CHG 的決策取捨,調和;必要時回 `modification-guide` 重做影響分析。

## 協調文件(並行時強烈建議)

`docs/coordination.md`(或一個看板):列出進行中的 claim——owner、鎖定範圍、CHG 號、狀態、開始時間。每個 agent 進出場都更新它。這是並行協作的「鎖表」。順序交接時可省略(用 CHG 狀態就夠)。

```markdown
# 協調看板
| Owner | 角色 | 讀寫權限 | 鎖定範圍 | CHG | 狀態 | 時間 |
|-------|------|---------|---------|-----|------|------|
| agentA | 實作者 | RW: src/modules/billing | src/modules/billing | CHG-20260616-03 | 進行中 | ... |
| agentB | 驗收者 | R only(唯讀) | src/modules/report | CHG-20260616-04 | 已完成 | ... |
```

**一 claim 一檔(原子性)**:多個 agent 同時改同一份 `coordination.md` 本身就有競態(兩者「同時」寫入、互相覆蓋)。真正並行時,建議**一個 claim 一個檔**——`docs/coordination/claims/<agent-id>.md`(檔案存在即上鎖;該檔發生 git 衝突 = 偵測到撞車)——`coordination.md` 保留為人讀的彙總。另注意 CHG 編號只在**同分支內**唯一(見 branch-isolation);不同分支上的 claim 不會互相佔號。

## 與既有流程的關係

- **順序交接** = 本 skill 的「Session 啟動檢查」+「當場驗收」的直接應用。
- **並行** = 在上述之上,再加 claim / 單寫者 / 整合驗收。
- **用哪一層握手**:由主 agent 派發的並行 agent 走**範圍握手**(派發者組包、審 ack);**沒有派發者的對等並行,各自做完整握手**——範圍進場的前提是有人為派發包負責(見 handshake「分層握手」)。
- 兩者都建立在「文件即真實 / 不倚賴記憶」之上——沒有可靠的共用文件,跨 agent 協作必然漂移。
