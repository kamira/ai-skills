# independent-acceptance — 跨 agent / 多情境獨立驗收

> 語言 / Language: **繁體中文** · [English](independent-acceptance.md)

## 用途

code 改完要驗收時,**不能由寫它的那個 agent 自己驗**——球員兼裁判,容易只測自己想到的 happy path、沿用自己當初的錯誤假設、對自己的盲點視而不見。本機制要求:由**不同 agent**、在**不同情境**下跑驗證,再彙整成一份 ACC。這是 acceptance-verification 階段的強化(獨立 + 多情境)。

> 「團隊」不限於人。**AI agent 團隊**同樣適用:實作 agent 與驗收 agent 是不同的執行個體/context,獨立性正來自此。

## 何時讀

- 一項變更(CHG)實作完成、要做驗收
- 需要比「自驗」更可信的把關(高風險變更、結構性變更)

## 依風險決定嚴格度

讀 CHG 的「風險分級」決定要多嚴:**高風險**→強制獨立驗收(驗收者 ≠ 實作者)+ 多情境;**中**→至少獨立或完整測試;**低**→可自驗。驗收結果寫進 ACC 時,**填上「驗收者」欄**(人 / agent id),讓「驗收者 ≠ 實作者」可被人或 CI 身分檢查驗證(見 ci-cd 的身分檢查 gate)。

## 核心原則:驗收者 ≠ 實作者

指派一個**獨立 agent** 做驗收,而且:
- **角色=驗收者,權限=唯讀**:驗收 agent 只能**讀**被驗的程式與條件,只能**寫**自己的 ACC 報告;**不可修改被驗的程式或結構文件**。一旦驗收者能改碼,就會「邊驗邊順手改」而失去獨立性、也可能埋新問題。發現問題→記進 ACC→回 `modification-guide` 由實作角色修,不是自己動手。
- 只給它**驗收條件來源**(Guideline §7 / 該 CHG 的目標與驗收條件)+ **成果(程式/系統)**。
- **不要**給它實作者的推理過程或「我覺得這樣對」的說法——避免被帶著走、複製同樣的盲點。
- 驗收條件回溯到**來源文件 / 使用者需求**,不是實作者的自述。

## 多情境驗證

同一條驗收條件,在不同情境下各驗一次,涵蓋面才夠:
- **不同輸入/邊界**:正常值、邊界值、異常值、空與極端。
- **不同環境**:乾淨環境重新安裝/建置後再跑測試(抓「在我機器上會動」類問題)。
- **不同切入點**:單元、API、端到端各驗,而非只靠一種。
- **不同 agent 分工**:必要時多個 agent 各驗一塊(如 A 驗功能正確、B 驗相容/回歸、C 驗非功能),再彙整。

## 防偏誤

- **自我確認偏誤**:自驗會傾向證明「它對」;獨立驗收傾向找「它哪裡會壞」。
- **共識偏誤(agent 團隊特有)**:若多個 agent 都基於同一份(可能錯的)假設或同一段被污染的 context,會「一起錯」。對策:讓驗收條件錨定在來源需求/文件,且至少一個驗收情境用乾淨 context(不帶實作對話)。
- **跨模型審查(破同模型盲點)**:同一個模型有相同的訓練偏誤與盲點——實作與驗收若用同一模型,某些錯會「系統性地一起看不到」。**驗收者盡量用與實作者不同的模型/供應商**(cross-model review);高風險變更尤其應如此。在 ACC 標明實作模型與驗收模型以利追溯。平台只有單一模型時,退而用「乾淨 context + 不同角色/system 設定」降低相關性,並在 ACC 註明此限制。
- **盲驗(進階)**:不告訴驗收 agent「預期答案」,只給條件與成果,讓它獨立判定。

## 彙整與收尾

- 多個獨立/多情境驗收結果**彙整成一份 ACC**:逐條列出每個情境的通過/未通過與證據。
- **任一情境未過 = 未通過**:回 `modification-guide` 修正(對應失敗回修迴圈),再重新獨立驗收。
- 全數通過才回填 CHG 狀態為「已驗收」並連結 ACC(對應「當場驗收不可交棒」)。

## 與既有流程的關係

- 延伸本 skill 的 `acceptance-verification`:同樣「對齊來源、每條附證據、可程式化就實測」,但把「誰來驗、在幾種情境驗」升級為**獨立 + 多情境**。
- 與 `cross-agent` 搭配:驗收 agent 透過 docs/ 取得驗收條件與成果現況,不需實作者口頭交接。
- 與 `ci-cd`(選用)搭配:多情境中「可自動化」的部分(乾淨環境、多輸入、回歸)交給 pipeline 反覆跑。
