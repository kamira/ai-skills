# structure-design — 產出四種結構文件

> 語言 / Language: **繁體中文** · [English](structure-design.md)

## 用途

把已確認的 AI Guideline 轉成可實作的系統結構,落成四份文件:**目錄結構、邏輯結構、設計結構、資料結構**。這些文件是後續實作與修改的藍圖,也是 `modification-guide` 評估變更影響、`acceptance-verification` 驗收的對照基準。

## 使用時機

- `docs/ai-guideline.md` 已確認,要開始規劃系統
- 使用者要求「設計架構」「規劃目錄」「設計資料庫/資料模型」「怎麼分層分模組」
- 重大重構前需要先確立目標結構

## 前置

先讀 `docs/ai-guideline.md`。結構必須回應 Guideline 的功能與非功能需求;若 Guideline 不存在或過時,先回到 `requirement-analysis`。

## 四種結構

四者由抽象到具體,彼此呼應。產出到 `docs/structure/` 下,各一份檔案。

### 1. 目錄結構 (directory.md)

專案實體的資料夾與檔案組織方式。

```markdown
# 目錄結構

## 樹狀
\`\`\`
project/
├── src/
│   ├── <模組A>/
│   └── <模組B>/
├── tests/
└── docs/
\`\`\`

## 各目錄職責
| 路徑 | 職責 | 備註 |
|------|------|------|
| src/<模組A> | ... | ... |

## 命名與放置規則
- <檔案/模組命名慣例、什麼東西該放哪>
```

### 2. 邏輯結構 (logical.md)

系統的分層與模組職責、模組間的相依與資料流(關注「做什麼、怎麼分工」,不綁實作細節)。

```markdown
# 邏輯結構

## 分層 / 模組
| 層/模組 | 職責 | 依賴 |
|---------|------|------|
| ... | ... | ... |

## 主要流程
<以文字或步驟描述 1-3 條關鍵流程,例如「使用者下單」從入口到完成的路徑>

## 相依方向
<誰依賴誰;應為單向、避免循環>
```

### 3. 設計結構 (design.md)

關鍵元件/介面/契約與採用的設計模式(關注「怎麼做、介面長怎樣」)。

```markdown
# 設計結構

## 關鍵元件
| 元件 | 責任 | 對外介面/契約 |
|------|------|----------------|
| ... | ... | ... |

## 介面 / API 契約
<重要介面的輸入、輸出、錯誤行為>

## 設計決策與取捨
| 決策 | 選項 | 取捨理由 |
|------|------|----------|
| ... | A vs B | ... |

## 採用模式
<採用的設計模式與原因,例如為何用 repository / event-driven>
```

### 4. 資料結構 (data.md)

資料實體、欄位、關聯與約束(資料庫 schema、主要資料物件)。

```markdown
# 資料結構

## 實體 / 資料表
### <實體名稱>
| 欄位 | 型別 | 約束 | 說明 |
|------|------|------|------|
| id | ... | PK | ... |

## 關聯
<實體間的關係:一對多、多對多、外鍵>

## 索引 / 約束
<重要索引、唯一性、刪除策略>

## 狀態 / 列舉
<關鍵狀態機或列舉值>
```

## 工作流程

1. 讀 Guideline,列出要被結構回應的需求(尤其 P0 與非功能需求)。
2. 由抽象到具體依序產出:邏輯 → 設計 → 資料 → 目錄(順序可依專案調整,但四份都要有)。
3. 確認四份彼此一致:邏輯的模組要能對到目錄的資料夾、設計的元件、資料的實體。
4. 在每份文件標註它回應了哪些需求編號(可追溯到 Guideline 的 FR/NFR)。
5. 交回使用者確認。

## 撰寫要點

- **通用、不綁特定技術棧**:除非 Guideline 已指定語言/框架/DB,否則用通用詞描述(層、模組、實體、介面),把實作選型留到設計決策表中說明理由。
- **可追溯**:結構元素盡量對應到 Guideline 的需求編號,日後驗收與變更才接得上。
- **保持一致**:四份文件描述的是同一個系統的不同視角,名稱與邊界要對得起來。
- **記錄取捨**:設計決策表是最有價值的部分——寫下為何選 A 不選 B,未來變更時才知道能不能動。

## 交付後

結構確認後即可實作。日後任何變更,交給 `modification-guide`,它會回頭更新這四份文件。
