## 動機
稽核發現:(1) marketplace 頂層 `metadata.version` 自建立起凍在 1.0.0,期間 plugin 版號已到 1.1.0/1.2.0——catalog 版本未動,client 認定目錄無變更、不重抓內部 plugin 新版,即「plugin 無法更新」的 repo 側主因;(2) ai-sdlc-suite plugin 版號 1.0.0→1.1.0 但無 `docs/ai-sdlc-suite/CHANGELOG.md`,plugin 層版本無對應記錄。使用者要求「之後每次 plugin 變動都同步 bump catalog」——散文規則在本 repo 已反覆被證明會被略過(見 ai-sdlc KN-001),故機械化為 CI 閘門。

## 決策與取捨
| 決策 | 選項 | 前提假設 | 為何這樣選 |
|------|------|----------|------------|
| catalog metadata.version=遞增計數,plugin/skill 變動即須 bump | 綁定=max(plugin) | max 綁定在「bumped plugin 未超過現有 max」時不會強制 catalog 動,仍會漏更新 | 計數語意直接對應「更新偵測」;git-aware --since 強制 |
| 機械強制=CI(catalog_check.py) | 只寫發佈步驟散文 | 散文期望在本 repo 必被略過(KN-001) | 靜態 --check(entry==plugin、semver)恆跑 + git --since(變動必 bump)PR 跑 |
| --since 無法解析 ref → exit 0(非誤殺) | 一律 fatal | 環境可能未 fetch base | 防 CI 環境性誤殺(前有 plan-check 誤殺前例);靜態檢查仍恆守 |
| catalog 現值 bump 到 1.2.0 | 1.1.0 | 已累積多次目錄變更;對齊最新 plugin 較不「落後」 | 1.2.0;此後每次 plugin release 再遞增 |
