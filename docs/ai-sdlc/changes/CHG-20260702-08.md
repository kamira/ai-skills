## 動機

(1) knowledge 單檔在規模化後成為寫入熱點、merge 衝突磁鐵、計數 diff 噪音源;INDEX 手維護必漂移。(2) AI-friendly 不只結構,還有語言:中文 tags 的分詞邊界傷檢索、規則行措辭不可測試就不可執行、跨模型(平台中立)以英文檢索鍵最穩。

## 決策與取捨

| 決策 | 選項 | 前提假設 | 為何這樣選 |
|------|------|----------|------------|
| 門檻 ~30 條才拆檔 | 一開始就拆 vs 門檻切換 | 小專案單檔+INDEX 最省 | 沿用歸檔/lite 的門檻哲學:規模到了才付結構成本 |
| INDEX=生成物 | 手維護 vs 腳本重生 | 手工複本必漂移 | `knowledge_index.py` 重生;`--check` 驗新鮮;lint 交叉驗條目↔INDEX id |
| tags 一律英文小寫固定詞彙 | 跟隨專案語言 vs 英文 | CJK 分詞傷 grep/檢索;跨模型英文最穩 | 檢索鍵英文;來源引文保留原語言(證據保真) |
| 規則行正規化 + 升級時做語言正規化 | 隨寫 vs 正規化 | 「盡量 X」不可測試=不可執行 | 祈使句、一條一義、可測試措辭;shallow→deep 升級儀式含正規化一次 |
| lint 只做輕量交叉檢查 | lint 內建重生比對 vs id 交叉 | 跨腳本 import 增加耦合 | doc_integrity 驗「條目檔↔INDEX id 雙向存在」;完整比對交給 knowledge_index --check |
