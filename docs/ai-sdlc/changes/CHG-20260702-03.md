## 動機

1. **UTC+0**:date-based ID、租約逾時、「同一天」序號在跨時區團隊會分裂;定一個鐘。
2. **分層握手**:完整握手不分對象會撞讀取預算,且 subagent 不需要全貌。採提案:scoped handshake 四鍵 = **branch + 結構位置 + 需求 + 鎖定位置**;不同分支不讀彼此、不同結構的 subagent 不讀彼此;完整視野與審查權collapse到派發的主 agent。

## 決策與取捨

| 決策 | 選項 | 前提假設 | 為何這樣選 |
|------|------|----------|------------|
| UTC+0 寫成全域慣例、模板標注 | 各專案自選時區 vs 統一 UTC+0 | 團隊可能跨時區 | 統一才能讓租約/ID/「同日」判定無歧義;模板標注讓人看得見 |
| scoped handshake 只給「被派發的 subagent」 | 所有 agent 都可 scoped vs 僅派發鏈 | 有人組包才有人負責漏切 | 對等並行無主 agent → 各自完整握手;責任不落空 |
| 全域 knowledge 穿透 scope | 完全隔離 vs 全域穿透 | 使用者 directive 常是通用禁忌 | 隔離掉全域規則=重犯已教過的錯,穿透是唯一例外 |
| 審查=四鍵比對 scoped ack | 免審 vs 派發者審 | ack 已是「正確接手」憑證 | 派發者有全貌,比對成本低;錯位在動手前攔下 |
| 代價明文化 | 隱含 vs 寫出 | scoped 必有盲區 | 寫明補償控制(影響分析/整合驗收/V1),避免誤以為 scoped=免責 |
