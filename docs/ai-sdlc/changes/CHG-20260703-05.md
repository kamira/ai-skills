## 動機

使用者四項:(1) **知識庫必須先建立**——專案一開始就建,連「使用者指示下達方式需要調整」這種互動層觀察也要入庫;(2) **主決策不參與用戶問答**;(3) **子 agent 代問**,需求完全了解後產出**企劃**交主 agent;(4) 主 agent 把企劃交**審查 agents 共同審議**。原理:問答者會被對話錨定(沉沒脈絡、附和、被帶著走)——裁決者只讀產物、不讀逐字稿,決策獨立性才保得住。這是「實作者不自驗」在需求端的鏡像:**引導者不裁決,裁決者不引導**。

## 決策與取捨

| 決策 | 選項 | 前提假設 | 為何這樣選 |

| 代問角色=A1(強化),不另設新角色 | 新設 E1 elicitor | A1 本就負責釐清與分析 | 職責歸一:A1 擁有「與使用者的需求問答」全部;不增角色數 |
| 主 agent 讀企劃不讀逐字稿 | 全部轉給主 agent | 逐字稿會複製錨定 | 與 independent-acceptance「不給驗收者實作者自述」同構 |
| 企劃交審議會共同審議(≥5 席) | 主 agent 單獨審 | v1.15 席次已定 | 決策層品質規則直接套用到需求企劃 |
| 確認閘仍直達使用者 | 一切經 A1 | 授權關係 ≠ 問答 | 「不參與問答」是資訊蒐集分工;最終核准權永遠是使用者對主 agent 流程的把關,不可轉包 |
| 知識庫先建=需求分析階段落地 | 首次糾正才建 | 空 INDEX 也是合法知識庫 | 第一天就有容器,互動風格觀察從第一句對話就能入庫(tag: interaction) |
| 無 spawn 降級 | 強制分離 | 單人環境沒有子 agent | 主 agent 自問並在記錄註明「引導與裁決同一人」的限制 |
