## 動機

markdown 條目靠 regex 抽欄位——中英欄名、全半形冒號、格式漂移全靠容錯,而**容錯就是誤讀的溫床**(讀錯比讀不到更危險:悄悄套用錯的規則)。JSON 解析二值:成功或大聲失敗,永不「讀成別的意思」。repo 既有哲學已是「JSON 為程式的單一真相、表格為人用視圖」(role_refs/halt_policy),knowledge 補齊同一原則。

## 決策與取捨

| 決策 | 選項 | 前提假設 | 為何這樣選 |
|------|------|----------|------------|
| JSON 而非 YAML/TOML | YAML 人類友善 | YAML 有 Norway problem 等隱性轉型 | 「不會讀錯」正是要排除隱性轉型;JSON 各語言原生、零縮排陷阱 |
| 拆檔模式正典=JSON;單檔 md 模式保留 | 全面強制 JSON | 小專案人讀優先、量少誤讀代價低 | 門檻哲學一致:規模到了才付格式成本;prospective 不追溯 |
| schema 檔+lint 手驗必填/enum | 引入 jsonschema 套件 | 腳本零依賴原則 | 手驗 required+enum+id=檔名,夠用且無新依賴 |
| 解析失敗=lint fail | 略過壞檔 | 壞檔=規則靜默消失 | fail-loud:寧可擋提交,不可悄悄少一條規則 |
| 人用視圖=生成的 INDEX.md | 另生 INDEX.json | AI 讀 md 表格無歧義風險(只是選單) | 誤讀風險在條目解析,不在選單;維持單一生成物 |
