## 動機（審視結論：knowledge 從未自動建立的根因鏈）

實證：本 repo 至 v1.14 已累積 13 筆 CHG、runner 專案多筆 CHG，兩邊 `docs/knowledge/` 皆不存在、零條目。根因（依貢獻度排序）：

1. **觸發點是散文、不是欄位**：modification-guide 第 7 步規定「收尾時比對動機是否重複前 CHG → 第二次出現建 shallow record」，但 CHG 模板**沒有對應欄位**——模型收尾時沒有結構性提示，既有 CHG 全數略過此步而無任何痕跡。
2. **零機械 enforcement**：`doc_integrity_check.py` 的 knowledge 檢查全部以 `docs/knowledge/entries/` **已存在**為前提（不存在→直接 return []）；`governance_health.py` 只統計 knowledge 數量，**零筆不產生任何警示**。缺目錄、缺檢查，一路靜默。
3. **沒有 bootstrap**：目錄從未誕生，下游機制全部 no-op。→ **上游 v1.16.0（CHG-20260703-05）已補新專案側**：需求分析建 `docs/` 時一併建 `docs/knowledge/`（空 INDEX 合法）。**但既有受治理專案（如 runner）的進場側仍缺**：handshake 第 3 步只「讀」INDEX（若存在），發現缺目錄不會補建。
4. **DIR 路徑依賴使用者明確糾正**：實務上使用者回饋多屬需求變更（正確走 CHG），從未被歸類為 correction directive——合法地未觸發，但也沒有任何條目能證明機制活著。

架構層面判定：四階段流程與文件驅動設計**符合現況**（v1.16.0 各宣稱物均存在且一致），但「自動治理」目前實質是「文件驅動的自律治理」——腳本閘門存在卻無處自動執行（兩 repo 均無 CI/pre-commit 接線；ci-cd.md 為 optional），且 knowledge 子系統從未運轉。v1.16 已修根因 3 的一半；本 CHG 修根因 1、根因 3 另一半、根因 2。CI 接線屬 ci-cd.md 範疇，維持 optional，不在本次範圍。
