## 動機
紀律 19-23 的每日操作有一半是確定性機械活(盤點誰、誰到期、誰連續無訊號、記錄處置),散文執行必漏。**明確排除**:定向搜尋/lookback 抓取/命中比對的資訊面——來源與判讀由使用者自理;runner 永不呼叫任何搜尋或網路。

## 決策與取捨
| 決策 | 選項 | 前提假設 | 為何這樣選 |
|------|------|----------|------------|
| runner 只管帳本側,不碰資訊獲取 | 內建 agent-cmd 搜尋 | 使用者明定來源自理 | 邊界乾淨:runner=狀態機與記錄員,判斷與資料是使用者的 |
| coverage 記錄=每日一 JSON(append-only) | 寫回快照/單一累計檔 | 快照不可覆蓋;計數要跨日可稽核 | `docs/intel/coverage/YYYY-MM-DD.json` 一日一檔,冷鏈計數由讀最近 N 檔推得 |
| tracking_status 可就地更新於 latest | 一律新快照 | Notion 原機制即允許驗證時就地填 latest | 機率/內容變更才需新快照;追蹤狀態轉換(休眠/喚醒)屬營運欄位 |
| 休眠只「建議」不自動改檔 | runner 直接改 | 降冷鏈是判斷(紀律 21 允許喚醒例外) | runner 列候選+理由,使用者確認後自行改或叫 agent 改 |
