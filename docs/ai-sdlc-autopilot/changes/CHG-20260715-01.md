- Skill: ai-sdlc v1.17

## 動機
autopilot 迴圈原本收尾=「逐 task TDD 測試綠 → 整支 review → ACC」。但 task 級 `test:` 是單元/build 級(RED-GREEN,證明零件對),**不證明變更真的跑起來、真的被操作過**——這正是「單元全綠但功能實際壞掉」的盲區(同 ai-regression-testing「同一模型寫又審」的失效模式)。IPB 式的「最後一哩」需要實際操作證據:跑起來、操作它、觀察行為。落地為新階段 operational_verify,插在整支 review 與 ACC 之間,並機械化為 runner 閘門(不是散文期望)。

## 決策與取捨
| 決策 | 選項 | 前提假設 | 為何這樣選 |
|------|------|----------|------------|
| operational_verify 為獨立階段,插在整支 review→ACC 之間 | 併入 task_review / 併入 acceptance | 操作驗收語意與逐 task 審查、與 ACC 文件產出都不同 | 獨立階段=可獨立設停點、可獨立豁免 docs-only |
| 閘門在 run 階段,plan-check 只提示不阻斷 | plan-check 也擋 | CI 只跑 plan-check;既有 CHG 無此欄不該被追溯 fail | run 時強制、CI 綠、既有記錄不受影響(同「新規則前瞻適用」) |
| 計畫加 `### Acceptance operation`(operate/observe/pass) | 沿用 task `test:` 欄 | 操作驗收是整支級、非 task 級 | 一個整支級宣告勝過散在各 task;缺它=程式變更收尾被擋 |
| docs-only 宣告可豁免 | 一律要求 | 純文件 CHG 無可操作面 | `Acceptance-operation: n/a (docs-only)` 一行豁免,不逼造假操作 |
| 高風險 operational_verify=halt(人執行) | 一律 auto | 高風險操作簽核不可機器自證 | 低/中=verify-cmd 可自動、無則人在迴圈;高=永遠人執行 |
