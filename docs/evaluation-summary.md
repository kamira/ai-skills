# ai-sdlc 評測總結(精簡)

說明用文件,非 skill 內容、不入封裝。彙整開發過程中的實證測試(原始逐項記錄在本機 gitignore 的 `skills/ai-sdlc-workspace/`,未入庫)。所有測試以子代理實跑(帶 skill vs 對照組),由人工 + assertion 評分。

## 一、基本效益(iteration 1–3)

| 測試 | 情境 | 帶 skill | 對照組 | 重點 |
|------|------|---------|--------|------|
| iteration-1 | 4 個獨立任務(需求/結構/修改/驗收) | 100% | 57% | 「會不會做」治理動作 |
| iteration-2 | 端到端複雜專案,對照真實 Medusa / Cal.com 架構 | 100% | 50% | 治理品質;產出貼近生產級架構 |
| iteration-3 | **單句目標、不列階段** | 100% | 17% | 不提示時是否自動帶出流程 |

結論:越不給提示,差距越大。對照組常直接傾印程式碼、零需求/結構/變更文件;帶 skill 則自動先做需求+結構+治理。iteration-2 的帶 skill 產出對上真實架構(Medusa 模組化+Promotion Module;Cal.com monorepo+round-robin)高度吻合。

## 二、累加需求 × session 模式(loop 測試)

主項目 + 依序加 R1 標籤 / R2 留言 / R3 子任務(結構性變更)。

| 條件 | 治理文件 | CHG | ACC | 觀察 |
|------|---------|-----|-----|------|
| 無skill·同session | 無 | 0 | 0 | 靠記憶維持一致,但零治理留痕 |
| 無skill·不同session | 無 | 0 | 0 | 靠讀碼重推,可跑但無制度記憶 |
| 現行skill·同session | 完整 | 3 | 4 | 完整治理鏈 |
| 現行skill·不同session | 完整 | 3 | **1** | 抓到破口:**跨 session 驗收懸空** |
| 修正後·不同session | 完整 | 3 | **4** | 修好:當場驗收 + Session 啟動檢查 |

**關鍵發現與修正**:跨 session 時「驗收被交棒卻沒人接」→ 加入「Session 啟動檢查」+「當場驗收不可交棒」,ACC 由 1 補回 4。
**壓縮補測**:同 session 在 context 壓縮下會退化成「只剩文件當記憶」;小而乾淨的專案兩組都還能靠讀碼救回,但治理軌跡只有 skill 留得下來 → 固化成原則「不倚賴記憶,以文件為準」。

## 三、團隊機制比對(team-test)

- **independent-acceptance(獨立驗收)**:同一份程式,獨立 agent 跑 16 情境 vs 實作者自驗 9 情境,**抓出自驗漏掉的時區 naive/aware crash 等真實風險**。價值明確。
- **doc-integrity(抗漂移)**:植入無聲漂移(程式有欄位、文件無),小專案下兩組都偵測到;skill 的價值在「定性 + 治理式處置(不可默默改、回流程留痕)」與規模放大後的可靠性。

## 四、代理編制實跑(hierarchy-run)

A1 分析 → I1 實作主(遞迴開 I1.1/I1.2 子代理 + 自檢)→ V1 獨立唯讀驗收。

- **權限分離在磁碟上成立**:A1 只寫 docs/、I1 鏈只寫 src/+CHG、V1 只寫 ACC(唯讀驗證),無人越權。
- **遞迴可行**:I1(本身是子代理)成功再開孫代理;權限只縮不放。
- **鐵律守住**:I1 不自驗、V1 不改碼、驗收條件回溯來源文件。
