# acceptance-verification — 驗收成果

> 語言 / Language: **繁體中文** · [English](acceptance-verification.md)

## 用途

在實作或修改完成後,把成果拿來對照「當初說好的標準」逐項查核,產出一份明確的驗收報告(通過/未通過/部分通過),並指出未達標項與後續處理。重點是**驗收標準來自既有文件,不是現場臨時發明**。

## 使用時機

- 一項功能或修改剛完成,要確認是否達標
- 使用者說「做完了/驗收/檢查一下/這樣對嗎」
- 一個 `modification-guide` 變更實作完成,要收尾
- 階段性交付前的把關

## 驗收基準來源

驗收條件按優先序取自:

1. 該次 `docs/changes/CHG-*.md` 的修改指引與目標(若為某次變更的驗收)
2. `docs/ai-guideline.md` 第 7 節「驗收條件」與功能/非功能需求
3. `docs/structure/*.md`(確認實作與結構一致)

若以上文件缺失,先回到對應 skill 補齊基準,再驗收;不要憑印象驗收。

## 工作流程

1. **蒐集基準**:列出本次該滿足的所有條件(逐條對應到需求編號或修改指引步驟)。
2. **逐項查核**:對每條,實際檢查成果是否符合,記下證據(指出哪個檔案/行為/測試證明達標)。可程式化驗證的(例如有無對應測試、欄位是否存在),寫腳本或實測,不要只用看的。
3. **判定**:每條標記 通過 / 未通過 / 不適用,並寫證據或落差。
4. **產出報告**:用下方模板寫入 `docs/acceptance/ACC-YYYYMMDD-NN.md`。
5. **處理未達標**:未通過項回饋給 `modification-guide` 產生修正,形成迴圈,直到全數通過或使用者接受。

## 累積迴歸集(保護舊功能)

單次驗收只回答「新東西做對了嗎」;光靠它,沒有任何機制回答「舊東西被弄壞了嗎」。所以驗收承諾要累積:

- **登記簿**:`docs/acceptance/regression.md`——歷次 ACC 中**可程式化驗證**的條件,一條一行:條件、怎麼跑(測試/腳本指向)、來源 ACC、範圍標籤(模組/區域)。執行指向用反引號包(如 `` `tests/test_x.py::test_y` ``)——lint 會驗指向的檔案仍存在;被刪掉的測試=靜默作廢的承諾。
- **沉澱**:每次 ACC 收尾時,把其中可腳本化的條件登記進去(不重複;範圍標籤用擴充的)。
- **執行**:**中/高風險**變更驗收時,跑登記簿中範圍標籤與本次 CHG 影響範圍相交的項目。**弄壞舊條件=未通過**,即使新條件全過——照一般未達標交回修正。低風險:跑影響範圍明顯碰到的即可。
- 登記簿是「指向」測試/腳本,不是重造測試框架——它的職責是讓歷來的驗收承諾保持可執行、而且真的被重跑。

## 驗收報告模板

```markdown
# ACC-YYYYMMDD-NN — <驗收標的>

- 專案:<專案代號 / 名稱>   ← 跨專案時必填,對應同專案的 CHG / Guideline
- 分支(Branch):<branch>   ← 多分支時必填;驗收條件只取同分支的 Guideline/CHG(見 branch-isolation)
- 日期:YYYY-MM-DD(UTC+0)
- 標的:<功能 / 變更編號 CHG-...>
- 驗收者:<人 / agent id>   ← 應 ≠ 該 CHG 的實作者(高風險變更必須不同;CI 可據此做身分檢查)
- 實作模型 / 驗收模型:<model A> / <model B>   ← 高風險宜跨模型(不同則記下,相同則註明限制)
- 風險分級:高 / 中 / 低(沿用對應 CHG;決定要驗幾種情境)
- 基準來源:<ai-guideline §7 / CHG-... 修改指引>
- Commit/PR:<hash / PR 連結>   ← 與 CHG 和程式碼同一 commit/PR(commit 錨定;見 modification-guide)
- Skill: ai-sdlc v<X.Y>   ← 本報告依哪版慣例寫成
- 結論:通過 / 部分通過 / 未通過

## 查核明細
| # | 驗收條件 | 來源 | 結果 | 證據 / 落差 |
|---|----------|------|------|-------------|
| 1 | ... | FR-1 | 通過 | <檔案/測試/行為> |
| 2 | ... | NFR-效能 | 未通過 | <落差描述> |

## 未達標項與處置
| 條件 | 落差 | 建議處置 | 交回 |
|------|------|----------|------|
| ... | ... | ... | modification-guide |

## 結構一致性檢查
- [ ] 實作與 docs/structure/ 一致(否則交回 modification-guide 更新)

## 迴歸(中/高風險)
- [ ] 已跑 docs/acceptance/regression.md 中受影響範圍的項目:<全綠 / 失敗項列於上方>
- [ ] 本 ACC 可程式化的條件已沉澱進迴歸集

## 總結
<整體是否可交付,還缺什麼>
```

## 撰寫要點

- **每條要有證據,且證據要可重跑**:「通過」要附指令+關鍵輸出,或審閱者能直接打開的檔案/行號指向——不是敘述性宣稱。沒有可重現查核的「通過」視同未驗證。
- **可程式化的就實測**:能用腳本/測試查的條件,實際跑,別目測。
- **未達標要可行動**:寫清楚落差與建議處置,並指向 `modification-guide` 形成修正迴圈。
- **檢查結構一致性**:不只看功能對不對,也看實作有沒有偏離結構文件;偏離就回 `modification-guide` 同步。
- **對齊來源、不擴張**:只驗當初說好的條件;若過程發現新需求,走需求/修改流程,別在驗收時夾帶。

## 交付後

全數通過則該階段完成,回填對應變更記錄的狀態為「已驗收」並連結本報告。有未通過項則回到 `modification-guide` 修正後重驗。
